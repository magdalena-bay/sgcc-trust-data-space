# Single Chain 8-Node Plan On One Server

更新时间：`2026-06-26`

这份文档回答的问题是：

`如何在当前这台单机服务器上，把后续链拓扑升级成“单条链 8 个节点”，以及这样做是否可行。`

先说结论：

`技术上可行，但它解决的主要是“拓扑与展示要求”，不是当前 10000+ 吞吐量的核心解法。`

也就是说：

1. `单链 8 节点`
   可以做
2. 但它不应被误认为：
   `吞吐量一定会因此暴涨`
3. 当前更适合把它当成：
   `比赛架构硬性要求 + 链拓扑扩展任务`

---

## 1. 当前真实现状

截至 `2026-06-26`，当前远端服务器是：

1. `8 vCPU`
2. `30 GiB RAM`
3. `15 GiB swap`

当前区块链实际运行状态是：

1. `qingdao`
   - `node0`
   - `node1`
2. `weifang`
   - `node0`
   - `node1`
3. `relay`
   - `node0`
   - `node1`

也就是：

`3 条链 x 2 节点 = 6 个 FISCO BCOS 节点`

当前目录结构在服务器上是：

1. `/home/ubuntu/blockchain/fisco/qingdao/127.0.0.1/node0`
2. `/home/ubuntu/blockchain/fisco/qingdao/127.0.0.1/node1`
3. `/home/ubuntu/blockchain/fisco/weifang/127.0.0.1/node0`
4. `/home/ubuntu/blockchain/fisco/weifang/127.0.0.1/node1`
5. `/home/ubuntu/blockchain/fisco/relay/127.0.0.1/node0`
6. `/home/ubuntu/blockchain/fisco/relay/127.0.0.1/node1`

所以现在并不是：

`单链 8 节点`

而是：

`三链各 2 节点`

---

## 2. 为什么要单链 8 节点

如果后续比赛要求是：

`单条链 8 个节点`

那它的意义主要有 3 类：

1. 展示网络规模
2. 展示共识网络完整性
3. 为后续多组织、多角色、多节点部署做拓扑基础

但它不天然等于：

1. 写交易 TPS 一定更高
2. 单机压力一定更小
3. 应用层就不需要做异步批处理

---

## 3. 单机 8 节点是否可行

结论：

`可行，但要受控部署。`

### 可行性的理由

1. 当前机器已经能稳定承载：
   `6 个链节点 + 3 个 WeBASE + MySQL + Redis + PostgreSQL + IPFS + privacy-service + platform-api`
2. 当前恢复后的常态内存占用大约只有：
   `3 ~ 4 GiB`
3. 当前 CPU 也没有持续打满
4. Swap 已经配置好

### 风险点

1. 单机 8 节点会增加：
   - 进程数
   - 端口数
   - 日志量
   - 数据目录
2. 如果仍然叠加：
   - GUI
   - vite
   - 自启开发态 watcher
   依然可能把机器重新压垮
3. 节点越多，单机上的“节点间网络”越像本地进程通信，展示价值大于真实分布式容灾价值

---

## 4. 设计原则

单链 8 节点方案必须遵守下面这些原则：

1. 不重启 GUI
2. 不恢复服务器端 `vite`
3. 不给前端 dev server 开任何常驻自启
4. 仍然优先用：
   `platform-api 托管静态前端`
5. 节点、自身脚本、WeBASE 都要手动受控启动
6. Verkle 底层逻辑不因节点数量变化而修改语义

也就是说：

`链节点数量变化，应该只影响链拓扑与部署，不应该反向破坏 Verkle 业务闭环。`

---

## 5. 推荐的最终链拓扑理解

如果后续必须同时满足：

1. 单链 `8` 节点
2. 仍然保留：
   `qingdao / weifang / relay`
   这样的业务分域表达

那需要先做一次架构取舍。

### 方案 A

保留三链结构，不马上改成单链 8 节点。

适合场景：

1. 当前业务逻辑高度依赖：
   `区域链 + 中继链`
2. 比赛短期更看重业务闭环

### 方案 B

把现有三链业务逻辑逐步抽象成：

`单链多组织 / 多分区 / 多合约命名空间`

然后在单链上做：

`8 节点`

适合场景：

1. 比赛后续硬性要求明确写成：
   `单条链 8 节点`
2. 允许业务语义从“三条物理链”迁移到“单条链上的多域建模”

### 当前建议

当前更建议：

`先把单链 8 节点作为独立部署实验与资源方案完成，不立刻强行迁移主业务链路。`

原因是：

1. 现在业务层已经真实跑通：
   `区域链 + 中继链`
2. 直接把主干从三链改成单链，会牵动：
   - 合约地址
   - chain registry
   - 审计解释逻辑
   - 文档叙事

这类改动现在做，收益不如风险小。

## 6. 单机 8 节点推荐部署方式

推荐部署结构：

`1 条链 + 8 个 FISCO BCOS 节点 + 1 套或少量 WeBASE 前端入口`

### 目录建议

建议单独新增，例如：

1. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node0`
2. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node1`
3. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node2`
4. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node3`
5. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node4`
6. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node5`
7. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node6`
8. `/home/ubuntu/blockchain/fisco/single8/127.0.0.1/node7`

这样做的好处是：

1. 不覆盖当前已跑通的三链目录
2. 便于随时回滚
3. 便于单独压测

### 端口建议

建议采用连续端口段，避免混乱，例如：

1. RPC / channel：
   `21000 ~ 21007`
2. P2P / listen：
   `22000 ~ 22007`

如果还需要管理端口或 exporter，再单独分配连续区间。

---

## 7. WeBASE 在 8 节点方案中的建议

不建议：

`8 个节点就配 8 套 WeBASE-Front`

原因：

1. 内存浪费
2. 管理复杂
3. 对你当前目标没有明显收益

建议做法：

1. 单链 8 节点下，优先只保留：
   `1` 套主 WeBASE-Front
2. 最多再配：
   `1` 套备用或对照用

这样能把 Java 常驻数量压住，避免再回到之前的资源风险。

## 8. 资源评估

### 8.1 粗略判断

在当前 `8 vCPU / 30 GiB / 15 GiB swap` 下，

`单链 8 节点`

从内存角度看大概率可做。

因为当前 6 节点全套业务都稳定时，整体使用量仍比较低。

### 8.2 仍需重点控制的部分

真正需要重点控制的不是链节点本身，而是：

1. WeBASE Java 进程数量
2. 是否再次出现 `vite/esbuild`
3. 是否重新启用 GUI
4. 日志滚动是否失控
5. 是否把节点都配成无边界自启

### 8.3 当前建议的资源策略

1. 链节点可以常驻
2. WeBASE 尽量减到最少
3. `platform-api`、`privacy-service` 仍优先手动受控启动
4. 前端继续只走静态托管
5. 不要把 8 节点实验和三链主业务同时长期全量常驻

也就是说：

`单链 8 节点实验环境` 和 `当前三链业务环境`

更适合：

`二选一切换运行`

而不是长时间全部一起开。

---

## 9. 对 Verkle 底层实现的影响

这一点必须明确：

`单链 8 节点部署，不会改变 Verkle 底层必须继续实现这一目标。`

原因是：

1. Verkle 的输入输出边界仍然是：
   `dataId / HD_i / proof / root`
2. 它依赖的是：
   `链上 root 语义`
3. 它不依赖当前一定要是：
   - 2 节点
   - 3 条链
   - 或 8 节点

所以节点数量变化不会推翻：

1. `VerkleEngineGateway`
2. `StoredProofEnvelope`
3. `proof envelope`
4. `MySQL / Redis / IPFS / root`
   这组闭环

后续真正需要调整的只是：

1. 链配置
2. 合约部署地址
3. chain registry
4. 业务如何解释“区域 root / 中继 root / 单链 root”

---

## 10. 现在是否应该立刻实施单链 8 节点

我的判断是：

`现在适合做设计、目录规划、端口规划、脚本规划，但不适合立刻把当前主干业务环境整体迁过去。`

原因：

1. 当前三链业务闭环刚稳定
2. 当前吞吐量 10000+ 方案还没进入链写入重构阶段
3. 现在最怕的是：
   一边改链拓扑，一边改业务主干，一边又冲吞吐量
4. 这样最容易把当前稳定面重新打散

当前更稳的做法是：

1. 先写清楚单链 8 节点部署方案
2. 再做一套独立脚本与目录
3. 单独起实验链
4. 最后再决定是否迁移主业务语义

---

## 11. 推荐执行顺序

1. 继续保持当前三链业务环境可用
2. 单独补：
   `single8`
   链目录与脚本
3. 先只起：
   `8 个 FISCO BCOS 节点`
4. 再接：
   `1 套 WeBASE`
5. 做基础连通性与资源验收
6. 再决定是否把：
   `anchorResource / recordAccess`
   测试迁过去
7. 最后才讨论业务层是否改成真正的单链语义

一句话结论：

`单机单链 8 节点在当前这台 8核30GiB 服务器上是可行的，但它主要解决的是链拓扑硬性要求，不是 10000+ 吞吐量的核心答案；Verkle 底层必须继续保留，而且不会因为节点扩成 8 个而被削弱，反而应该继续作为链下证明与链上 root 锚定之间的核心结构。`
