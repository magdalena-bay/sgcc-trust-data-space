# On-Chain Throughput 10000+ Plan

更新时间：`2026-06-26`

这份文档只回答一个问题：

`在不破坏当前业务逻辑和 Verkle 底层实现目标的前提下，怎样把“只测区块链上链环节时间”的技术路线设计到 10000+。`

先说结论：

`当前系统已经能稳定跑通 Verkle-compatible 全链路，但按现有同步单笔上链架构，不可能直接靠小修小补达到 10000+。`

所以这里给的不是“马上改两行代码就到 10000+”，而是：

`一条不推翻现有业务语义、可分阶段落地的架构升级路线。`

---

## 1. 当前测量口径

这里严格采用你指定的口径：

`只测区块链上链写交易本身的时间`

不计入：

1. IPFS 上传
2. MySQL 写入
3. Redis proof 写入
4. Verkle proof 重建
5. 解密
6. 完整上传与访问编排

当前代码里对应的链上写交易入口是：

1. `BlockchainGatewayService.anchorResource(...)`
2. `BlockchainGatewayService.recordAccess(...)`

底层实际调用路径是：

`WeBASE-Front /trans/handle`

---

## 2. 当前实测基线

截至 `2026-06-26`，当前小样本串行测试结果如下。

### 2.1 anchorResource

`qingdao`

1. `avg_ms = 1349.56`
2. `throughput_tps = 0.74`

`weifang`

1. `avg_ms = 1348.45`
2. `throughput_tps = 0.74`

`relay`

1. `avg_ms = 1330.43`
2. `throughput_tps = 0.75`

### 2.2 recordAccess

`relay`

1. `avg_ms = 1282.58`
2. `throughput_tps = 0.78`
3. `status = 0`

### 2.3 这组数字的准确含义

这组结果只能理解为：

`当前 2 节点/链 + WeBASE HTTP 写交易 + 串行单笔提交 模式下，单条链的单笔同步写入 TPS 大约只有 0.74 到 0.78。`

它不能被误读成：

`整个平台所有业务最多只有 0.74 TPS`

也不能被误读成：

`再多加几台节点就会自然到 10000+`

---

## 3. 为什么当前架构到不了 10000+

### 3.1 当前瓶颈不是 Verkle 本身

当前瓶颈主要不在：

1. `HD_i`
2. proof envelope
3. Redis proof 存储
4. MySQL 记录结构

当前瓶颈主要在：

1. `单笔同步写交易`
2. `HTTP -> WeBASE -> 链`
3. `每次业务动作都直接等待链上确认`

### 3.2 当前吞吐量为什么这么低

原因叠加如下：

1. 每次只发 `1` 笔交易
2. 每次都走 `WeBASE-Front /trans/handle`
3. 当前测试口径是：
   `提交一笔 -> 等返回 -> 再提交下一笔`
4. 当前单条链只有 `2` 个节点
5. 当前业务上传还带有：
   `同一区域 ACTIVE 资源共享当前 root`
   的语义约束

所以现在这套路径本质上是：

`低并发、强同步、单笔确认优先`

这和：

`10000+`

本来就不是一个量级的架构目标。

---

## 4. 不破坏现有业务逻辑的约束条件

这部分非常关键。

后续任何吞吐量升级，都不能破坏下面这些现有语义：

1. `dataId -> HD_i`
   这条资源标识与内容摘要关系不能变
2. Verkle 仍然必须存在
3. `proof envelope`
   这层抽象不能消失
4. MySQL / Redis / IPFS / 链上 root 的审计闭环不能丢
5. 前端、后端、审计接口的业务 DTO 不应大面积返工
6. 不能重新引入：
   `vite 常驻`
   `GUI 依赖`
   `无限自启`
   这类风险

换句话说：

`要改的是“上链写入管线”，不是把整个业务闭环推翻重来。`

---

## 5. 推荐的 4 阶段技术路线

## 5.1 阶段 A：先把“同步单笔直写链”抽象成受控写入层

目标：

`先不追求 10000+，先把业务层和链写入层解耦。`

建议做法：

1. 新增：
   `OnChainWriteGateway`
   或
   `OnChainWriteDispatcher`
2. 业务层不再直接关心：
   `是不是立刻同步提交到 WeBASE`
3. 业务层只投递：
   - `ANCHOR_RESOURCE`
   - `RECORD_ACCESS`
   - `ANCHOR_REGION_ROOT`
   这类标准化写入事件

这一步的意义是：

`先把“怎么上链”从“业务本身”里抽出来。`

### 对现有逻辑的影响

1. 不改现有前端接口
2. 不改 MySQL 主表模型
3. 不改 Redis proof envelope
4. 不改 Verkle 引擎边界

---

## 5.2 阶段 B：把“每条资源都直接上链”改成“事件缓冲 + 批次聚合上链”

这是最关键的一步。

目标：

`让 10000+ 的目标从“单笔同步交易 TPS”转成“业务事件摄入 TPS”。`

建议做法：

1. 上传事件先落：
   MySQL 业务表
2. 同时写入：
   `onchain_outbox`
   或
   `onchain_event_queue`
3. 后台受控 worker 按时间窗或条数窗聚合
4. 聚合后只把：
   - 一批资源的摘要集合
   - 一批访问审计的摘要集合
   - 当前批次的 Verkle root / batch root
   上链

### 为什么这不破坏现有逻辑

因为现有业务真正需要的是：

1. 可审计
2. 可证明
3. 可验证
4. 可解释

而不是：

`每一个上传动作都必须立即单独形成一笔同步链交易`

### 这里对 Verkle 的要求

Verkle 不能删。

相反，它要继续保留并强化为：

1. 资源层：
   `dataId -> HD_i -> proof -> root`
2. 批次层：
   `batchId -> batchRoot`
3. 链上层：
   `链上锚定 batchRoot / regionRoot / relayRoot`

这意味着：

`Verkle 底层不仅不能去掉，反而会成为高吞吐量方案里的关键聚合抽象。`

---

## 5.3 阶段 C：绕开 WeBASE HTTP 单笔同步瓶颈

如果目标真的是：

`10000+`

那单靠现在的：

`WeBASE-Front /trans/handle`

很难完成。

后续建议路线：

1. 保留 WeBASE 作为调试与管理入口
2. 高吞吐量写入改走：
   - 更贴近链 SDK 的客户端
   - 异步发送
   - 多连接复用
   - 批量发送
3. 压测时区分：
   - `交易提交 TPS`
   - `交易确认 TPS`
   - `业务事件摄入 TPS`

### 为什么不建议现在马上大改

因为当前项目的首要目标仍然是：

1. 业务闭环稳定
2. Verkle 底层继续落地
3. ABE / DID / AI Agent / 联邦学习继续推进

所以这一层更适合作为：

`链写入子系统专项重构`

而不是现在直接把主业务代码到处改散。

---

## 5.4 阶段 D：链上存“批次承诺”，明细走链下可验证审计

这是最接近 `10000+` 目标的真正落点。

推荐最终口径改成：

`业务事件摄入吞吐量 >= 10000`

而链上承担：

1. 批次 root
2. 批次摘要
3. 审计窗口摘要
4. 关键索引锚点

这样做的好处是：

1. 业务事件仍然可审计
2. Verkle 证明仍然存在
3. 链上数据量显著下降
4. 不需要每个动作都做同步单笔确认

---

## 6. 推荐的目标分层

为了避免以后指标口径打架，建议把吞吐量拆成 3 层。

### 6.1 当前口径

`单笔同步链上写交易 TPS`

当前大约：

`0.74 ~ 0.78 TPS`

### 6.2 近期可追求口径

`批次提交后的链上确认 TPS`

目标建议：

`100 ~ 500+`

这取决于：

1. 批次大小
2. 并发发送
3. 合约结构
4. 单机资源

### 6.3 最终比赛叙事口径

`业务事件摄入 TPS`

目标建议：

`10000+`

但必须明确前提：

`这是建立在批次聚合锚定与链下可验证明细审计之上的体系指标，不是“每个业务动作单独同步上链 10000 TPS”。`

---

## 7. 与当前代码的兼容改造点

推荐改造集中在下面几层：

1. `BlockchainGatewayService`
   收口成：
   `同步直写实现`
   和
   `异步/批次实现`
   两种策略
2. `DemoResourceService`
   继续保留当前业务语义，但不直接承担高吞吐发送职责
3. 新增：
   - `OnChainOutboxEntity`
   - `OnChainBatchEntity`
   - `OnChainDispatchWorker`
4. 合约层增加：
   - 批次摘要锚定接口
   - 批次访问审计摘要接口

这些改动不会要求：

1. 前端整体返工
2. Verkle 引擎推倒重来
3. ABE / DID / Agent / FL 模块暂停

---

## 8. 当前不建议做的事

1. 不建议现在为了追 `10000+` 直接把上传接口改成“完全不校验、不入库、只打日志”
2. 不建议现在把 Verkle 删掉，只保留链上哈希
3. 不建议把所有链写动作都粗暴合并成一个全局 root，而失去资源级审计能力
4. 不建议重新把服务器端前端切回 `vite` 常驻
5. 不建议为了压测方便，重新打开 GUI 或引入不受控自启

---

## 9. 对比赛叙事的建议表达

后续如果需要向老师、评委或队友解释“为什么当前是 0.74 TPS，但目标写 10000+”，建议统一这样说：

1. 当前实测的 `0.74 ~ 0.78 TPS`
   是：
   `单笔同步链上锚定写交易吞吐量`
2. 后续目标 `10000+`
   不是靠“每个业务动作逐笔同步上链”达成
3. 后续将通过：
   `Verkle 批次承诺 + 事件缓冲 + 聚合锚定 + 异步提交`
   达成高吞吐摄入
4. 这条路线不破坏现有业务语义，也保留 Verkle 作为底层可验证结构

---

## 10. 当前建议的执行顺序

1. 继续保持当前 Verkle-compatible 全链路稳定
2. 先完成：
   `OnChainWriteDispatcher / Outbox / Batch`
   方案设计
3. 再做：
   `批次锚定合约接口`
   设计
4. 然后再做：
   `高吞吐量专项压测`
5. 最后才把比赛指标口径收口到：
   `业务事件摄入 10000+`

一句话结论：

`当前系统已经具备继续开发的稳定底座，但“只测链上写交易时间”的 10000+ 目标，必须通过链写入架构升级来实现；Verkle 底层不是障碍，反而应该被保留并升级为批次聚合与可验证审计的核心抽象。`
